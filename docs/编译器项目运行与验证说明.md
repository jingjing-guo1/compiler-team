# 编译器项目运行与验证说明（正常情况）

以下描述假设编译器已经完整实现了词法分析、语法分析、语义分析、中间代码生成和最终代码生成（即成员1至6全部完成）。这些命令是验证编译器正确性的标准流程。

## 1. 拉取最新代码
`git pull origin main`  
正常情况：该命令会从 GitHub 仓库的 main 分支下载团队最新的提交，输出类似“Already up to date.”或显示更新的文件列表。这样能保证你使用的是团队最新版本。

## 2. 编译编译器
`"D:\software\mingw64\mingw64\bin\gcc.exe" -Iinclude -o compiler.exe src\*.c`  
正常情况：没有任何错误或警告，并生成 `compiler.exe` 可执行文件。如果路径中 gcc 已在环境变量，也可以直接使用 `gcc -Iinclude -o compiler.exe src\*.c`。

## 3. 确认编译器已生成
`dir compiler.exe`  
正常情况：列出 `compiler.exe` 文件，文件大小不为 0，且日期为当前编译时间。

## 4. 查看帮助信息
`compiler.exe --help`  
正常情况：输出所有支持的命令行选项及其用途，例如：
```
Usage: compiler.exe [options] <input.c>
Options:
  -o <file>     Output assembly file (default: output.s)
  -v            Verbose output
  --dump-tokens Print token list
  --dump-ast    Print abstract syntax tree
  --dump-ir     Print intermediate code
  -h, --help    Show this help
```

```
# 解释：
#   -o         : 指定生成的汇编文件名（默认 output.s）
#   -v         : 详细模式，打印编译各阶段信息
#   --dump-tokens : 输出词法分析得到的 Token 序列（验证词法）
#   --dump-ast    : 输出抽象语法树（验证语法）
#   --dump-ir     : 输出中间代码（验证中间代码生成，成员5的任务）
#   --help     : 显示本帮助信息
```



## 5. 使用 `--dump-tokens` 查看词法分析结果

`compiler.exe --dump-tokens test\factorial.c`  
正常情况：逐行输出源代码中的每个词法单元（Token），包括类型和文本值。例如对于 `int main()` 会输出：
```
KEYWORD_INT    "int"
IDENTIFIER     "main"
LPAREN         "("
RPAREN         ")"
...
```
没有“非法字符”或错误提示。

## 6. 使用 `--dump-ast` 查看抽象语法树
`compiler.exe --dump-ast test\factorial.c`  
正常情况：输出树状结构的语法树，清晰地显示函数定义、语句、表达式等。例如 factorial 函数的 AST 会包含函数声明、参数、if 语句、返回语句等节点。输出中不应出现 “syntax error”。

## 7. 使用 `--dump-ir` 查看中间代码
`compiler.exe --dump-ir test\factorial.c`  
正常情况：输出三地址码（或四元式）形式的中间表示，例如：
```
FUNC_ENTER factorial
  t0 = n == 0
  IF t0 THEN L1
  t1 = n * factorial(n-1)
  RETURN t1
L1:
  RETURN 1
FUNC_EXIT
```
代码逻辑正确，没有重复块或死循环。

## 8. 正常编译（生成汇编代码）
`compiler.exe test\factorial.c`  
正常情况：不输出任何错误信息，并生成 `output.s` 汇编文件。该文件内容为合法的汇编代码（如 x86-64 或 ARM），可以通过 `gcc output.s -o factorial` 链接成可执行文件，运行后输出正确的计算结果。

## 9. 验证语义分析（错误处理）
`compiler.exe test\semantic_errors.c`  
正常情况：编译器中会一次性报告多个语义错误（例如变量未声明、类型不匹配、重复定义等），但不会崩溃或生成汇编代码。输出格式清晰，统计错误总数。

## 总结
以上所有命令的正常表现就是编译器团队最终应该达到的目标。
