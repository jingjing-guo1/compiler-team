# 编译器项目 —— TINY 语言编译器

本仓库是《编译原理》课程设计“TINY 语言编译器”的完整实现。

项目目标：为 TINY 教学语言（教材《编译原理及实践》中的示例语言）开发一个可生成 x86-64 汇编代码的编译器。编译器采用经典的“词法分析 → 语法分析 → 语义分析 → 中间代码生成 → 目标代码生成”架构，支持赋值、条件、循环、输入输出等语言特性，并提供清晰的模块化接口、Git 协作流程以及**真实编译器风格的多错误收集与恢复机制**。

## 功能范围

编译器当前支持以下 TINY 语言特性：

- 变量声明（隐式声明，由赋值引入）
- 赋值语句 `变量 := 表达式`
- 条件语句 `if 表达式 then 语句序列 end` 以及 `if ... then ... else ... end`
- 循环语句 `repeat 语句序列 until 表达式`
- 输入语句 `read 变量`
- 输出语句 `write 表达式`
- 算术表达式（`+` `-` `*` `/`）和关系表达式（`<` `=`）
- 注释 `{ ... }`
- 整型常数和标识符

编译输出为 x86-64 汇编代码（Intel 语法），可在 Linux 环境下使用 GCC 链接为可执行文件。

**待扩展**（根据课程进度可选）：
- 函数调用
- 数组支持
- 常量折叠等简单优化

## 多错误收集与恢复（核心特性）

本编译器采用**工业级错误处理策略**，模仿真实编译器行为：

- **词法错误**：遇到非法字符（如 `@`、`#` 等）时，报告错误并跳过该字符，继续扫描后续代码，一次性报告所有非法字符。
- **语法错误**：通过同步记号（分号 `;`、`end`、`repeat`、`if`、`read`、`write` 等）跳过一段输入，尝试恢复解析，使编译器能够发现多个语法错误而非遇错即停。
- **语义错误**：继续分析其他语句，一次性报告所有未定义变量、类型不匹配等问题。
- **错误上限**：默认最多报告 20 个错误（可通过修改 `g_state.max_errors` 调整），超过则停止编译，避免无意义输出。
- **最终输出**：只要存在任何错误（错误计数 > 0），就不会生成汇编代码，并输出错误摘要。

> 这种设计使得调试效率大大提高，用户无需反复编译即可看到所有明显错误。

## 技术路线

```
TINY 源代码 (.sy)
        ↓
词法分析器 (lexer.c) → Token 序列（非法字符跳过）
        ↓
语法分析器 (parser.c) → 抽象语法树 (AST)（同步点恢复）
        ↓
语义分析器 (semantic.c) → 带类型标注的 AST + 符号表（多错误收集）
        ↓
中间代码生成器 (irgen.c) → 三地址码 (四元式)（仅当无错误时）
        ↓
目标代码生成器 (codegen.c) → x86-64 汇编 (.s)（仅当无错误时）
        ↓
gcc 链接 → 可执行程序
```

技术栈：

| 层次       | 技术                        |
| ---------- | --------------------------- |
| 编程语言   | C11                         |
| 构建系统   | Makefile                    |
| 版本控制   | Git + GitHub                |
| 代码组织   | 模块化（每个阶段独立 `.c`） |
| 目标平台   | Linux x86-64（系统调用）    |
| 汇编语法   | Intel 语法（gcc 兼容）      |

## 快速启动

### 环境要求
- Linux 系统（或 WSL / macOS，但汇编输出依赖 Linux syscall，建议使用 WSL2）
- gcc、make

### 克隆仓库
```bash
git clone https://github.com/jingjing-guo1/compiler-team.git
cd compiler-team
```

### 编译编译器
```bash
make clean
make                # 编译 release 版
make debug          # 编译 debug 版（带符号）
```
成功后生成可执行文件 `compiler`。

### 运行编译器并查看多错误报告
```bash
./compiler -v test/minimal.sy
```

常用命令行选项：
- `-v`：打印详细进度信息（包括错误计数）
- `--dump-tokens`：输出 Token 列表
- `--dump-ast`：输出抽象语法树
- `--dump-ir`：输出中间代码四元式
- `-o <file>`：指定输出汇编文件名（默认 output.s）

### 测试多错误收集
你可以故意构造一个包含多个错误的源文件，例如：
```
x := 5;
y := x + ;     # 缺少右操作数
z := @;        # 非法字符
write y
```

编译器将一次性报告：
- 语法错误：表达式不完整
- 词法错误：非法字符 '@'
- 并继续扫描，最后输出错误摘要，不生成汇编。

## 协作规范

为确保代码质量和团队协作顺畅，我们采用以下 Git 工作流：

### 分支保护
- `main` 分支受保护，**任何人（包括组长）都不能直接推送**到 `main`。
- 所有变更必须通过 Pull Request（PR）合并。

### 开发流程
1. 从最新的 `main` 创建功能分支，命名格式 `feature/模块名`（例如 `feature/lexer`）。
2. 在分支上开发，实现对应的函数。
3. 提交代码并推送到远程仓库：
   ```bash
   git push -u origin feature/模块名
   ```
4. 在 GitHub 网页上发起 PR，目标分支 `main`。
5. PR 中应写明改动内容、测试情况、是否依赖其他模块的接口变更。
6. 组长（成员1）进行代码审查并批准（至少 1 人批准）。
7. 批准后由组长合并 PR，并删除该功能分支。

### 公共头文件更新
- `common.h` 由成员1统一维护。
- 如需修改（例如新增 Token 类型、修改函数原型），成员应先在 GitHub 上提 Issue 或在群中沟通，由成员1评估后修改并推送。
- 所有成员在开发前应执行 `git pull origin main` 确保使用最新的 `common.h`。

### 禁止事项
- ❌ 直接 `push` 到 `main` 分支（会被服务器拒绝）。
- ❌ 私自修改本地的 `common.h` 并提交（必须通过成员1）。
- ❌ 在 PR 中混入与当前功能无关的代码。

### 代码审查要点
- 接口是否符合 `common.h` 中的声明。
- 是否正确处理错误情况（报告错误但不立即停止，除非严重）。
- 是否有内存泄漏。
- 是否添加了必要的注释。

## 质量要求

1. **不虚构输出**：编译器只能根据输入的源代码生成正确的汇编代码，不能输出与源代码语义不符的指令。
2. **多错误报告**：应尽量收集并报告多个错误，而不是遇错即停。错误总数达到上限或发生严重错误时方可停止。
3. **错误恢复**：语法分析器应实现同步点机制，避免因一个语法错误导致后续大量无意义错误。
4. **主分支保持可编译**：任何时候 `main` 分支上的代码都应能通过 `make` 成功编译。
5. **模块独立**：每个模块只依赖 `common.h` 定义的数据结构和函数原型，不直接调用其他模块的内部实现。

## 常见问题

### Q：为什么编译器有时会报告多个错误但仍然继续运行？
**A**：这是设计目标。编译器会尽量收集所有明显错误（最多20个），提高调试效率。只有当错误数量超过上限或发生致命错误（如内存不足）时才会停止。

### Q：如何关闭多错误收集，改为遇错即停？
**A**：可以在 `main.c` 中将 `g_state.max_errors` 设置为 1，或在每次调用 `report_error` 后立即检查并 `exit`。但默认行为更符合真实编译器习惯。

### Q：修改了 `common.h` 后其他成员编译报错？
**A**：提醒成员1更新 `common.h` 并推送，其他成员执行 `git pull origin main` 同步。

### Q：汇编代码在 macOS 上无法运行？
**A**：本项目的汇编输出使用 Linux 系统调用（`syscall` 编号），若要在 macOS 上运行，需要修改系统调用号和指令（例如改用 `mov eax, 0x2000001` 等）。建议使用 Linux 或 WSL2 环境。
